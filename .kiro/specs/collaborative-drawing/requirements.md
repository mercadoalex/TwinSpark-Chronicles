# Requirements Document

## Introduction

Collaborative Drawing enables both siblings (Ale & Sofi) to draw and doodle together during story moments in Twin Spark Chronicles. At specific narrative beats, the Story_Engine presents a shared canvas where both children can draw simultaneously via WebSocket-synced strokes. Completed drawings are incorporated into the story narrative by the AI, creating a deeply personal and immersive experience where the children's art shapes the adventure. Drawings are persisted and visible in the storybook gallery alongside scene images.

## Glossary

- **Drawing_Canvas**: The shared HTML5 Canvas component rendered in the frontend that accepts touch/mouse input from both siblings simultaneously
- **Stroke**: A single continuous drawing path consisting of an array of coordinate points, a color, a line width, and the sibling identifier (child1 or child2)
- **Drawing_Session**: A time-bounded collaborative drawing period triggered by a story beat, containing all strokes from both siblings on a single canvas
- **Drawing_Sync_Service**: The backend WebSocket handler that receives, validates, and broadcasts stroke data between connected siblings in real time
- **Drawing_Store**: The Zustand store managing local canvas state, stroke history, undo stack, and sync status on the frontend
- **Drawing_Persistence_Service**: The backend service responsible for saving completed drawings as PNG images and storing metadata in the database
- **Story_Engine**: The existing orchestrator and storyteller agents that generate narrative content and scene images
- **Drawing_Prompt**: A narrative cue generated by the Story_Engine that tells the siblings what to draw (e.g., "Draw the magic door you want to open!")
- **Composite_Image**: The final PNG rendering of the completed collaborative drawing, saved to the assets directory
- **Storybook_Gallery**: The existing gallery feature where completed stories and their assets are displayed

## Requirements

### Requirement 1: Drawing Canvas Rendering

**User Story:** As a sibling, I want a large, touch-friendly drawing canvas so that I can doodle easily with my fingers on a tablet or with a mouse.

#### Acceptance Criteria

1. WHEN a Drawing_Session is active, THE Drawing_Canvas SHALL render a full-width canvas element with a minimum touch target area of 300×300 CSS pixels
2. THE Drawing_Canvas SHALL support simultaneous multi-touch input so that both siblings can draw at the same time on a shared device
3. THE Drawing_Canvas SHALL provide a color palette with at least 8 child-friendly colors, each color swatch having a minimum tap target of 44×44 CSS pixels
4. THE Drawing_Canvas SHALL provide at least 3 brush sizes (thin, medium, thick) selectable via tap-friendly controls
5. THE Drawing_Canvas SHALL assign a distinct default color to child1 strokes and a different distinct default color to child2 strokes so that each sibling's contributions are visually distinguishable
6. WHEN the viewport width is less than 768 pixels, THE Drawing_Canvas SHALL stack the tool palette below the canvas instead of beside it

### Requirement 2: Real-Time Stroke Synchronization

**User Story:** As a sibling, I want to see my sibling's drawing appear on my screen in real time so that we feel like we are creating together.

#### Acceptance Criteria

1. WHEN a sibling draws a stroke on the Drawing_Canvas, THE Drawing_Sync_Service SHALL broadcast that stroke to the other connected sibling within 100 milliseconds
2. THE Drawing_Sync_Service SHALL transmit stroke data as JSON messages over the existing WebSocket connection used by the Story_Engine
3. WHEN a stroke message is received from the Drawing_Sync_Service, THE Drawing_Canvas SHALL render the remote stroke using the originating sibling's color and brush size
4. IF the WebSocket connection is lost during a Drawing_Session, THEN THE Drawing_Store SHALL queue outgoing strokes locally and replay them upon reconnection
5. THE Drawing_Sync_Service SHALL validate that each incoming stroke message contains a valid session identifier, sibling identifier, and at least one coordinate point before broadcasting

### Requirement 3: Drawing Session Lifecycle

**User Story:** As a sibling, I want the story to tell us when to draw and when drawing time is over so that drawing feels like a natural part of the adventure.

#### Acceptance Criteria

1. WHEN the Story_Engine generates a story beat with a drawing prompt, THE Drawing_Canvas SHALL appear with an animated entrance transition and display the Drawing_Prompt text
2. THE Drawing_Session SHALL enforce a configurable time limit between 30 and 120 seconds, with a default of 60 seconds
3. WHILE a Drawing_Session is active, THE Drawing_Canvas SHALL display a visual countdown indicator showing remaining time
4. WHEN the Drawing_Session time limit is reached, THE Drawing_Canvas SHALL disable further input and display a completion animation
5. THE Drawing_Canvas SHALL provide a "We're Done!" button that allows siblings to end the Drawing_Session before the time limit
6. WHEN a Drawing_Session ends, THE Drawing_Canvas SHALL exit with an animated transition and return focus to the story display

### Requirement 4: Drawing Persistence and Composite Image Generation

**User Story:** As a parent, I want my children's collaborative drawings saved so that we can revisit them in the storybook gallery.

#### Acceptance Criteria

1. WHEN a Drawing_Session ends, THE Drawing_Persistence_Service SHALL render all strokes into a single Composite_Image in PNG format
2. THE Drawing_Persistence_Service SHALL save the Composite_Image to the assets/generated_images directory with a filename following the pattern drawing_{unix_timestamp}.png
3. THE Drawing_Persistence_Service SHALL store drawing metadata in the database including the session identifier, sibling pair identifier, stroke count, drawing duration in seconds, and the file path to the Composite_Image
4. IF the Composite_Image rendering fails, THEN THE Drawing_Persistence_Service SHALL log the error and continue the story without interruption
5. THE Drawing_Persistence_Service SHALL associate the Composite_Image with the corresponding story beat so that the Storybook_Gallery displays the drawing alongside the scene

### Requirement 5: AI Narrative Integration

**User Story:** As a sibling, I want the story to react to what we drew so that our doodles feel like they matter in the adventure.

#### Acceptance Criteria

1. WHEN a Drawing_Session completes, THE Story_Engine SHALL receive a description of the drawing context (prompt text and sibling identifiers) and incorporate a narrative acknowledgment into the next story beat
2. THE Story_Engine SHALL generate the next story beat narration referencing the collaborative drawing activity within 5 seconds of the Drawing_Session ending
3. WHEN generating the post-drawing story beat, THE Story_Engine SHALL include personalized references to both siblings by name in the narration

### Requirement 6: Drawing Tools — Undo and Eraser

**User Story:** As a sibling, I want to undo my last stroke or erase parts of the drawing so that I can fix mistakes without starting over.

#### Acceptance Criteria

1. THE Drawing_Canvas SHALL provide an undo button that removes the most recent stroke drawn by the tapping sibling
2. WHEN the undo button is tapped, THE Drawing_Store SHALL remove the last stroke from the active sibling's stroke history and re-render the canvas
3. THE Drawing_Canvas SHALL provide an eraser tool that paints with the canvas background color at the selected brush size
4. THE Drawing_Store SHALL maintain a separate undo stack per sibling so that one sibling's undo does not affect the other sibling's strokes
5. IF the undo stack for a sibling is empty, THEN THE Drawing_Canvas SHALL visually disable the undo button for that sibling

### Requirement 7: Drawing Data Serialization

**User Story:** As a developer, I want stroke data serialized in a well-defined JSON format so that it can be transmitted, stored, and replayed reliably.

#### Acceptance Criteria

1. THE Drawing_Sync_Service SHALL serialize each stroke as a JSON object containing fields: session_id (string), sibling_id (string), points (array of {x, y} objects), color (hex string), brush_size (integer), and timestamp (ISO 8601 string)
2. THE Drawing_Sync_Service SHALL deserialize received JSON stroke messages back into stroke objects and validate all required fields are present
3. FOR ALL valid stroke objects, serializing then deserializing then serializing SHALL produce an identical JSON string (round-trip property)
4. IF a received stroke message fails validation, THEN THE Drawing_Sync_Service SHALL discard the message and log a warning without disrupting the Drawing_Session

### Requirement 8: Drawing Canvas Accessibility

**User Story:** As a parent of a child with motor difficulties, I want the drawing tools to be accessible so that all children can participate.

#### Acceptance Criteria

1. THE Drawing_Canvas SHALL support keyboard-based color and tool selection using arrow keys and Enter
2. THE Drawing_Canvas SHALL announce tool changes and drawing session state transitions via ARIA live regions
3. WHILE a Drawing_Session is active, THE Drawing_Canvas SHALL maintain a minimum color contrast ratio of 3:1 between each palette color and the canvas background
4. THE Drawing_Canvas SHALL provide a "stamp" mode with pre-made shapes (star, heart, circle, lightning bolt) as an alternative to freehand drawing for children who find precise drawing difficult

### Requirement 9: Session Time Limit Respect

**User Story:** As a parent, I want drawing time to count toward the session time limit so that my children do not exceed their allowed screen time.

#### Acceptance Criteria

1. WHILE a Drawing_Session is active, THE Session_Time_Enforcer SHALL continue counting elapsed time toward the session time limit
2. IF the session time limit is reached during a Drawing_Session, THEN THE Drawing_Canvas SHALL immediately end the Drawing_Session and trigger the standard session-end flow
3. WHEN the remaining session time is less than the Drawing_Session time limit, THE Story_Engine SHALL reduce the Drawing_Session duration to fit within the remaining session time
